# o1 / o1-mini — Unofficial Tips

Tips specific to `o1`, `o1-mini`, and `o1-preview`. These are community-sourced and complement the [official guidance](../../docs/openai/o1.md).

---

## Prompting Style

### Be terse — o1 rewards brevity
o1's internal reasoning is thorough; a long prompt with meta-instructions ("first consider X, then Y, then Z") adds noise without adding value. State the problem once, clearly.

```
✅  "Prove that √2 is irrational."
❌  "Think carefully. Consider all possible approaches. Systematically reason through each step before arriving at your conclusion. Prove that √2 is irrational."
```

### Provide constraints, not instructions about how to reason
Tell o1 *what* answer you need and *what constraints* it must satisfy — not *how* to think.

```
✅  "Return a solution in O(n log n) time using only standard library functions."
❌  "Think about the time complexity first, then consider which data structures help..."
```

### State ambiguities explicitly
o1 will make assumptions to resolve ambiguity. If you want a specific assumption, state it. Otherwise, ask o1 to list the assumptions it makes.

```
"Assume the input array is zero-indexed. List any other assumptions you make."
```

## Task Selection

### When to use o1 vs GPT-4o
| Task | Recommended model |
|------|------------------|
| Multi-step mathematical proof | o1 |
| Competitive programming (hard problems) | o1 |
| Complex legal / scientific reasoning | o1 |
| Code generation (standard complexity) | GPT-4o |
| Summarisation | GPT-4o |
| Casual Q&A | GPT-4o |
| Quick classification | gpt-4o-mini |

### Use o1-mini for iterative development
o1-mini is significantly cheaper and faster than o1. Use it during prompt development; switch to o1 for final production runs on hard problems.

---

## `reasoning_effort` Tuning

| `reasoning_effort` | When to use |
|--------------------|-------------|
| `"low"` | Problems you expect o1 to solve easily; rapid iteration |
| `"medium"` | Default; balanced cost and quality |
| `"high"` | Hard competition-level problems; final production run |

Tip: Start with `"medium"`, check whether the answer is correct, then increase to `"high"` only if it fails.

## Output Format

### Ask for a brief explanation if needed
o1 suppresses its internal reasoning. If you want the reasoning in the output, ask explicitly:

```
"Solve the problem. Then provide a concise explanation of your approach."
```

This gives you a human-readable trace without asking the model to "show its work" step by step (which degrades quality).

### Use strict JSON schema for structured output
Same as GPT-4o — `response_format: json_schema` with `strict: true` works well with o1.

## Known Quirks

- **No streaming of reasoning tokens**: o1 streams only the final answer. Latency can be higher than GPT-4o for hard problems. Set client timeouts accordingly.
- **Higher latency at `reasoning_effort: "high"`**: can take 30–90 s for very hard problems. Inform users if building a real-time interface.
- **Does not benefit from few-shot examples on pure reasoning tasks**: for math/logic, few-shot examples sometimes hurt because they constrain the model's internal search. Test both zero-shot and few-shot.
- **May refuse ambiguous tasks more readily**: o1 is cautious; if it refuses, clarify the task rather than rephrasing to bypass safety measures.
